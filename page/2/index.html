我所理解的实例 状态 和 上下文

发表于 2017-02-13   |   分类于 programming   |     |   阅读次数

我所理解的对象和状态和上下文

对象

所谓对象,使用类型创造出来的实例,instance ,我们也经常用 instance 来表示,
构建一个 instance,然后操作 instance 的状态

对象最开始创造的时候,是在内存中的,根据业务需求,经过我们业务流程逻辑操作之后,
状态变更完成后,平常的操作是持久化到数据库中,当然也可以直接写到文件里,或者
经过网络发送,或者就放到内存里

当我们下次在使用某一个对象时,针对持久化到数据库的情况,在程序中构建一个空白的对象,
然后从之前选择的持久化方式中在将对象 load 出来,赋值给这个空白对象,就又还原了状态,
又可以根据业务需求,做响应的逻辑操作

其实这里可以想到整个对象的变化过程,理解平常自己干的工作的本质

状态

对象的状态,也可以理解成对象的值,比如一个 User 对象,如果 name 属性是 favoorr, 当前的
值就是 favoorr,别人使用这个对象的时候,值就是 favoorr,当然这个值可以修改,就是修改对象
的状态,多线程修改下还是很容易出问题的

上下文

平常经常使用的是 context,可以理解为某一个对象的管理范围,一个对象给你构造的是一个世界,
获取这个世界里的任何东西,都可以理解成从上下文获取

说的有点悬了,来个具体的例子

一个操作系统上面,运行了一个 Tomcat 容器,容器内运行了一个 webapp 监听 8080 端口,
处理用户请求

操作系统启动了

目前的计算机都是状态机,一个操作系统软件在硬件上启动后,就会初始化自己的状态,构建自己
上下文,所有构件好之后,操作系统有自己的状态,例如当前的端口使用情况,当前运行的进程数,
当前内存的使用量,这些从整个操作系统的上下文中都可以去获取,首先要知道,这些都会有,至于
自己会不会去获取这是另外一回事

进程启动了

然后安装了 Java 运行环境,启动了 Tomcat,启动了一个 Java 进程,初始化了 webapp,全部启动
完成后,webapp 监听 8080 端口处理用户请求

整个过程中,操作系统的状态改变了,内存进程用了,端口被用了,进程增加了

Tomcat 进程完成了初始化,有了自己的状态,而且可以通过很多命令来获取当前的状态

状态重要有了,就看应用是否愿意让你获取,开发者是否用心给你提供了方便查询的接口

用户请求来了

当一个用户在浏览器访问 localhost:8080/index.jsp 时,通过层层处理,到了 Tomcat 进程,这个进程
就开始处理了,在处理的过程中,有个 Servlet 的抽象,这个请求进来后,构建了一个实例,可以观察
各种状态,例如请求地址,请求 ip,都可以获取,同时这些所有对象都活在当前的 Tomcat 容器
上下文中

进程内根据自己的处理,返回请求数据后,请求结束了,这个请求构建的对象生命周期结束了

总结下

这些概念,如果自己理解了,可以去描述计算机的状态机,可以去猜很多东西,有些东西自己
不知道,但是一定有,可以直接去查,去搜,缩小自己不知道的不知道,迎接更大的不知道
