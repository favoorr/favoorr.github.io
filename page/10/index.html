maven使用总结

1.本地无法更新jar包了

有时候我们用私服，然后发现本地无法更新了，不会自动去下载jar，

先检查maven配置是否正确，
包括pom文件中Repository的配置和setting中mirror的配置，

然后如果无误，看私服服务器是否能连接外网

然后都检查过了，还是不行，就去把maven本地仓库中该jar的目录先删除，在下载，

还是不行就手工下载jar，放到目录下

2.m2e 有个offline ，有次无意间点到，彻底完蛋，麻痹的，不去访问私服，不能发布

3.用户目录下的 setting.xml 文件，如果本地仓库地址修改了，更改没问题，默认的应该是${user.home}/.m2/repository
自己曾经在mac上又一次把目录写为 ~/.m2/repository，原本以为能够自己解析，后来发现，我想错了，事情不是那样子
项目每次简单的构建都会下载全部的jar，浪费了很多时间，但是自己却不知道问题出在哪里,而且还不知道怎么去问别人
直到我看到每次构建都会在当前目录下新增 ~ 目录，而且新增jar包，我才意识到，本地仓库目录搞错了，我恍然大悟，
oh my gold ！

4.聚合和继承，是可以通过一个parent搞定，所有子模块依赖于 parent，然后可以选择性的快速构建

5.maven项目从 svn 或者 git check out 下来后，如果无法按照自己想要的目录结构呈现，可以从 Eclipse 里面删除项目，然后在重新选择导入
Exsting maven project可以搞定，或者点击项目名–>鼠标右键Configure–>选择convert to maven project，Convert稍微好用点

6.在使用maven-release-plugin执行release:prepare的时候，出现如下错误

/bin/sh: svn: command not found

解决方法

sudo ln -s /usr/local/bin/svn /bin/svn

7.maven 单独使用clean compile的时候不会报错，maven-release-plugin执行release:prepare的时候
出现如下错误，

无效的目标版本： 1.7

或者

无效的目标版本： 1.8

或者英文是

invalid target release: 1.7

invalid target release: 1.8

环境上我的mac上安装了 1.6 1.7 1.8 三个版本的jdk，JAVA_HOME设置如下：

export JAVA_HOME=`/usr/libexec/java_home -v 1.8`

Google之后，在stackoverflow找到的答案原因是你在maven配置的某个地方配置了固定版本的jdk，
但是你的maven-compile-plugin配置使用的是如下配置

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.1</version>
    <configuration>
        <source>1.6</source>
        <target>1.6</target>
    </configuration>
</plugin>

或许里面的1.6一开始写成了1.7或者1.8，然后就会报上面那个错误

解决的方法就是换不同的版本试试，1.6，1.7或者1.8，maven执行一些shell命令的时候，走得是 /bin/sh,
但是我默认的shell是zsh，这里不知道怎么设置，比如在使用maven-release-plugin时候，执行的内嵌shell
都是用 /bin/sh 执行的通过debug日志，可以很清楚看到

8.eclipse console中文乱码

解决方法是Run Configurations,找到对应的项目，在Common面板中，Encoding修改为GBK就ok了
可能是内置了GBK编码，但是默认mac上是 UTF-8

9.Maven 项目不显示 Maven Dependencies

在使用Eclipse开发maven项目时，有时候会碰到依赖jar没办法显示的问题，

选择项目右键  --> maven --> disable maven nature,可以手工重现

修复方法

选择项目右键 --> Configure  --> Convert to maven project

9.当两个项目引用的 jar 包 artifactId 和 version 相同，遇到的诡异问题

今天业务开发同学来找到我说 guava-18.0.jar 这个包有问题，说是在本地打的包没错，
到测试环境上这个包总是错的，坐标如下

然后自己本地 md5 hash 947641f6bb535b1d942d1bc387c45290 ,跟 nexus 是一样的，
但是到测试上的包就有问题，告诉我是 jenkins 上的 guava 包有问题，后来我下载 jenkins 上
打的包，果然 md5 值不是 nexus 上的值，公司的 jenkins 是我维护的，我到 guava 目录看了下包
hash 是没错的，最后发现是某个组件依赖了一个其他的 guava，group id 不同，但是 artifactId 和 version
都是相同的，结果就会出现这种问题，jenkins 上有一段提示信息

可以看到在打包的时候，这个包被跳过了，这里就是问题所在，而且这种问题相当难发现

36     version "3.8.5.0"
37     sha1 "c5655a4004095c50dc8403661e0ed02fd4191d57"
38   end

brew install --universal --java subversion

参考链接http://stackoverflow.com/questions/25904364/subclipse-error-svn-sqlite-compiled-for-3-8-6-but-running-with-3-8-5

mac上JavaHL报错解决

发表于 2014-12-01   |   分类于 svn   |     |   阅读次数

描述

升级svn到1.8.10后，打开Eclipse进入显示以下异常：

Failed to load JavaHL Library.
These are the errors that were encountered:
no libsvnjavahl-1 in java.library.path
no svnjavahl-1 in java.library.path
no svnjavahl in java.library.path
java.library.path = /Users/User/Library/Java/Extensions:/Library/Java/Extensions:/Network/Library/Java/Extensions:/System/Library/Java/Extensions:/usr/lib/java:

原因

javaHL 版本和subversion版本和subeclipse版本是由对应关系的，详细参见连接http://subclipse.tigris.org/wiki/JavaHL
根据提示，执行如下操作

brew install --universal --java subversion

出现错误

Warning: subversion-1.8.10_1 already installed

然后先卸载subversion，

brew uninstall subversion

重新安装

brew install --universal --java subversion

同时下载1.10.6版本的subeclipse，地址：http://subclipse.tigris.org/archive/

链接方式安装插件后，重启Eclipse，JavaHL没有报错

mac os x使用Apache与svn集成

发表于 2014-12-01   |   分类于 svn   |     |   阅读次数

写在前面

mac os x使用Apache server与svn集成，提供http访问svn，下面是详细步骤，中间有遇到的各种问题
及解决方法.

首先说既然要配置http方式访问svn，那我就默认大家已经配置了svn 本地仓库，
并且已经可以使用类似svn://127.0.0.1/account/trunk这种方式访问成功，还没有配置
的，可以参照这里

以下为配置主要步骤：

  1. 手工编译svn，获取依赖包mod_authz_svn.so、mod_dav_svn.so，Apache server需要的，但是
    mac上通过homebrew安装的svn是没有这些包得，原生的Apache server也没有

  2. 修改Apache server配置文件/etc/apache2/httpd.conf，加载对应的module

  3. 建立svn.conf配置文件，添加Apache server解析svn配置

  4. 重启Apache server

  5. 使用浏览器访问，验证配置

下面为详细配置

下载，手工编译

svn的源码包下载地址http://subversion.apache.org/download/

编译安装
./configure --with-apxs=/usr/sbin/apxs
make

注意这里不需要安装，如果仅仅这样安装了版本，是不支持http和https方式访问的

浏览是否编译成功
ll subversion/mod_authz_svn/.libs/mod_authz_svn.so
ll subversion/mod_dav_svn/.libs/mod_dav_svn.so

复制到Apache server的lib目录
sudo cp subversion/mod_authz_svn/.libs/mod_authz_svn.so /usr/libexec/apache2/
sudo cp subversion/mod_dav_svn/.libs/mod_dav_svn.so /usr/libexec/apache2

检查
ll /usr/libexec/apache2 | grep svn

安装后，Apache server下应该有以下lib
/usr/libexec/apache2|grep dav
mod_dav.so
mod_dav_fs.so
mod_dav_lock.so
mod_dav_svn.so

修改apache配置文件

sudo vim /etc/apache2/httpd.conf 

添加以下配置,左侧为行数
152 LoadModule dav_module libexec/apache2/mod_dav.so
153 LoadModule dav_fs_module libexec/apache2/mod_dav_fs.so
154 LoadModule dav_svn_module libexec/apache2/mod_dav_svn.so
155 LoadModule authz_svn_module libexec/apache2/mod_authz_svn.so

建立svn配置文件

<Location /svn>
    DAV svn
    SVNParentPath /Users/User/Documents/svn-repos
    SVNListParentPath on
    AuthType Basic
    AuthName "Subversion repositories"
    AuthUserFile /Users/User/Documents/svn-repos/account/conf/passwd
    AuthzSVNAccessFile /Users/User/Documents/svn-repos/account/conf/authz
    Require valid-user
</Location>

SVNParentPath 为svn仓库的根目录，参照/Users/User/Documents/svn-repos/account/conf/passwd,
看下自己该输入哪个路径

AuthUserFile 为svn密码管理的路径

AuthzSVNAccessFile 为权限配置的路径

添加svn用户

根据提示输入密码
sudo htpasswd -c /Users/User/Documents/svn-repos/account/conf/passwd admin

这里注意，这种添加的用户密码就不是明文了，和之前直接修改添加的不同

配置权限

vim /Users/User/Documents/svn-repos/account/conf/authz

测试

打开浏览器访问地址
http://localhost/svn

如果配置正确，能够看到正确地svn项目列表

总结

我在安装的时候遇到了很多问题，我首先google的关键字是mac Apache svn，然后
发现配置需要的module没有，然后手工编译svn，

编译svn的时候，我的是1.8.*版本，最后我用的是1.8.10，又下载了一系列的依赖包，而且
1.7.* 和 1.8.* 提供http和https访问使用的还不同

最后还是被我编译成功了，但是问题又出现了，在跟Apache server集成的时候，出现错误

httpd: Syntax error on line 120 of /etc/apache2/httpd.conf: 
Cannot load /usr/libexec/apache2/mod_dav_svn.so into server: 
dlopen(/usr/libexec/apache2/mod_dav_svn.so, 10): 
Library not loaded: /usr/local/bin/apache2/lib/libaprutil-1.0.dylib\n  
Referenced from: /usr/local/lib/libsvn_repos-1.0.dylib\n  
Reason: Incompatible library version: libsvn_repos-1.0.dylib requires version 6.0.0 or later, 
but libaprutil-1.0.dylib provides version 4.0.0

最后解决的方法我在这篇文章中受到启发http://www.dssw.co.uk/blog/2014-01-07-serving-svn-with-httpd-on-os-x-10-9/

在解决的过程中，发现我之前自己编译安装的svn会给通过brew安装造成很多错误，
最好先清空下列文件夹中svn开头的内容

/usr/local/include
/usr/local/lib
/usr/local/share/man/man1/
/usr/local/share/man/man5/
/usr/local/share/man/man8/

直到brew install subversion成功，

启动Apache server后，又遇到错误

apache 日志目录 /private/var/log/apache2/

[Sat Nov 29 21:15:00.773969 2014] [auth_basic:error] [pid 45831] [client 127.0.0.1:54634] AH01617: 
user favoorr: authentication failure for "/svn/account": Password Mismatch

我的路径/Users/User/Documents/svn-repos/account/ Documents,需要读写权限

chmod -R 755 /Users/User/Documents

然后重启Apache server访问才成功.
