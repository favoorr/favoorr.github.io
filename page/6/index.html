网络-分析

netstat

多种网络栈和接口统计信息

默认 列出连接的套接字
-a 列出所有套接字的信息
-s 网络栈统计信息
-i 网络接口信息
-r 列出路由表

其他选项能修改输出，例如

-n 不解析 ip 地址为主机名
-v 显示冗长的详细信息

Iface 网络接口
MTU 最大传输单元
RX 接收
TX 传输
OK 成功传输的数据包
ERR 错误数据包
DRP 丢包
OVR 超限

丢包和超限是网络接口饱和的指针

-c 连续模式能与 -i 一并使用，每秒输出这些累积的计数器，它提供技术数据包速率的数据

网络栈统计数据

输出列出了多项按协议分组的网络数据主要是来自 TCP 的

sar

统计信息历史

可以观测当前活动病情能配置为保存和报告历史统计数据

rxpck/s
txpck/s
rxkB/s
txkB/s

rxcmp/s

rxerr/s 接收数据包错误
txerr/s 传输数据包错误
coll/s 碰撞
rxdrop/s 接收丢包 缓冲满
txdrop/s 传输数据包丢包 缓冲满
rxfifo/s 接收的数据包 FIFO 超限错误
txfifo/s 传输的数据包 FIFO 超限错误

irec/s
fwddgm/s
orq/s

idisc/s
odisc/s

active/s 新的主动 TCP 连接 connect()
passive/s 新的被动 TCP 连接 listen()
iseg/s 输入的段（接收）
oseg/s 输出的段 (传输)

retrans/s TCP 段重传

totsck 使用中的总数据包 sockets
ip-frag 当前队列中的 IP 数据片 fragments
tcp-twTIME-WAIT 的 TCP 套接字 sockets

许多统计信息包括方向和统计单位

rx 是接收
i 输入
seg 是段

还可以统计 ICMP UDP NFS 和 IPv6 在内的信息
IpInReceives = irec/s

ifconfig

接口配置

列出所有接口当前的配置，与 netstat -i 一致
txqueuelen 是这个接口发送队列的长度

linux 下的 ifconfig 命令逐渐被 ip 命令替代

ip

网络接口统计信息

能够配置网络接口和路由，并且观测他们的状态和统计信息。

nicstat

网络接口吞吐量和使用率

-z 用来忽略数值为 0 的接口
-t 显示 TCP 统计信息

最前面是系统启动以来的总结，紧接着是按时间间隔的总结

Int 接口名称
%Util 最大使用率
Sat 反应接口饱和度的统计信息
r 读（接受）
w 写 （传输）

KB/s 千字节每秒
PK/s 数据包每秒
Avs/s 平均数据包每秒，以字节为单位

ping

测试网络连通性

rtt 每个包的往返时间
由于 时间戳是 ping 命令自己计量的，其中包括获取时间戳到处理网络 IO 的整个 CPU 代码路径执行时间

traceroute

测试网络路由

发出一系列数据包实验性地探测到一个主机当前的路由
它的实现利用了递增每个数据包 IP 协议的生存时间 TTL，
从而导致网关顺序的发送 ICMP 超时响应报文，向主机揭示自己的存在（如果防火墙没有拦截他们）

每一跳显示连续的三个 RTT，它们可用作网络延时统计信息的粗略数据源。
类似 ping 由于发送低优先级的数据包，它可能显示出比其他应用程序协议更高的延时

也可以把显示的路径作为静态性能调优的研究对象。网络被设计为动态的并且能响应故障。路径的变化可能会降低性能。

pathchar

确定网络路径特征

pathchar 类似于 traceroute ，并且包括了每一跳间的贷款，它的实现是利用了多次重复发送一系列不同长度的网络数据包然后统计分析它们

在 ubuntu 上，pathchar 叫 pchar

依据跳跃数的不同，可能需要数分钟

tcpdump

网络数据包嗅探器

可以捕捉并检查网络数据包。它输出网络包信息到 STDOUT，或者把数据包写入文件以供稍后的分析。后者通常更实用：过高的数据包速率导致了不能实时研究他们

将 eth4 的数据写入 /tmp 下的文件

每一行显示数据包时间（精度到毫秒），它的源和目标 IP 地址，以及 TCP 报头值。

-n 禁用 IP 地址解析为主机名
-v 打印包冗长的细节
-e 链路层报头
-x -X 十六进制地址转储

在性能分析过程中，把时间戳列改为显示数据包间的时间差 -ttt ，或者自第一个数据包以来的时间 -ttttt 会更有帮助

就 CPU 成本和存储而言，捕捉数据包是昂贵的，在可能的情况下尽量短时间使用 tcpdump 以限制其对性能的影响

wireshark

图形化网络数据包检查器

DTrace

TCP/IP 栈跟踪，连接、数据包、丢包、延时

perf

TCP/IP 栈跟踪，连接、数据包、丢包、延时

其他工具

strace 跟踪套接字相关的系统调用并检查其使用的选项，使用系统开销较高
lsof 按进程 id 列出包括套接字细节在内的打开的文件
ss 套接字统计信息
nfsstat nfs 服务器的和客户机统计信息
iftop 按主机嗅探总结网络接口吞吐量
/proc/net 包含许多网络统计信息文件

试验工具

iperf

iperf 是一款测试最大 tcp 和 udp 吞吐量的开源工具，必须同时在服务器和客户机上运行

服务器运行

将套接字缓冲长度由默认的 8KB 增加到 128KB (-1 128k)

客户机运行

最后会给一个总的汇总

-c host 连接到主机名或者 IP 地址
-1 128k 使用 128KB 套接字缓冲
-P 2 运行于两个客户机线程的并行模式
-i 1 每秒打印时间间隔总结
-t 60 总测试时间 60s

测试完成后，可以在 server 端看到结果

以 Ubuntu 为例列举 Memory 性能分析常用命令

发表于 2017-02-09   |   分类于 performance   |     |   阅读次数

操作系统版本 Ubuntu 16.04.1

手工实践记录，把新知识手工敲一遍，是把知识变成自己的第一步

内存分析工具

vmstat

虚拟和物理存储器统计信息
虚拟内存统计信息指令。提供了包括当前内存和换页在内的系统内存健康程度总览。
CPU 统计信息页包含在内

该版本的 vmstat 不在第一行输出自系统启动至今的 memory 列的汇总数据，而立刻显示当前状态。

默认数据列如下，单位为 KB

swpd 交换出的内存量
free 空闲的可用内存
buff 用于缓冲缓存的内存
cache 用于页缓存的内存
si 换入的内存 （换页）
so 换出的内存 （换页）

如果 si 和 so 一直非 0，那么系统郑村镇内存压力并换页到交换设备或文件，用其他工具可以研究什么在消耗内存，例如能观察每进程内存使用的工具

当服务器拥有大量内存是，数据列会不对齐而影响阅读。可以试着用 -Sm 来修改输出单位为 MB

选项 -a 可以输出非活动和活动页缓存的明细

sar

历史统计信息

-B 换页统计信息
-H 大页面统计信息
-r 内存使用率
-R 内存统计信息
-S 内存空间统计信息
-W 交换统计信息

pg 表示页
kb 表示 KB
% 表示百分比
/s 表示每秒

详细部分可读源码 mm/vmscan.c

%vmeff 是衡量页回收率的一个有趣指标。高数值意味着成功从非活动列表中回收了页，表示健康
低数值意味着系统在挣扎。 100% 是高数值，少于 30% 是低数值

slabtop

内核块分配统计信息

可以通过 slab 分配器输出内核 slab 缓存使用情况。类似 top 实时更新屏幕

输出包括顶部的汇总和 slab 列表，

OBJS 对象数量
ACTIVE 多少是活动的
USE 使用百分比
OBJ SIZE 对象大小
CACHE SIZE 缓存大小

sc 按缓存大小排序，最大值在顶端
slab 统计信息取自 /proc/slabinfo, 也可以用 vmstat -m 输出

ps

进程状态命令 ps 可以列出包括内存使用统计信息在内的所有进程细节

%MEM 主存使用（物理内存、RSS）占总内存的百分比
RSS 常驻集合大小 KB
VSZ 虚拟内存大小

RSS 显示主存使用，它也包括如系统库在内的映射共享段，可能会被几十个进程共享。如果把 RSS 列求和，可能会发现它超过系统的内存综合，这是由于重复计算了这部分共享内存。分析共享内存的使用可参考 pmap 命令

linux 下也可以输出 maj_flt min_flt 列

maj_flt 严重缺页
min_flt 轻微缺页

top

监视每个进程存储器使用率

监控排名靠前的运行中进程并且显示内存使用统计信息

Mem 主存
Swap 交换内存
buffers 缓冲缓存
cached 页缓存

VIRT 虚拟内存
RES 主存，常驻集合大小
%MEM 占用主存占比

pmap

进程地址空间统计信息

列出进程的内存映射，显示它们的大小、权限及映射的对象。这使进程内存使用能被仔细的检查以及量化共享内存。

Kbytes 虚拟内存
RSS 主内存
Dirty 私有匿名内存
mode 权限

很少有内存是匿名的，而且大部分是只读的 r-x，这意味着这些页面可以被其他进程共享，尤其是系统库。

对于有很多映射的进程，pmap 输出可能会很长。它还会暂停进程以报告内存使用，这会影响任务的性能。应在需要针对和分析时运行，不应作为监视工具频繁运行

DTrace

分配跟踪

DTrace 能跟踪用户和内核级的内存分配、轻微和严重缺页以及页面换出守护进程运行。这些功能支持使用特征归纳和想下挖掘分析。

其他工具

free 报告空闲内存，包括缓冲区高速缓存和页缓存
dmesg 检查来自 OOM 终结者的 Out of memory 信息
valgrind 一个包括 memcheck 在内的性能分析套件，它是一个内存使用分析的封装程序，可用于发现泄露。它能造成严重的系统开销，可能引起目标系统慢 20 - 30 倍
swapon 添加和观察物理交换设备或者文件
iostat 如果交换设备是物理磁盘或块，设备 I/O 可以用 iostat 来观测，它能指出系统是否在换页
perf 能观察 CPI MMU TSB 事件，以及源自 CPU 性能计数器的内存总线的停滞周期计数。还提供缺页以及一些内核内存 kmem 事件的探针
/proc/zoneinfo 内存区域的统计信息
/proc/buddyinfo 内核页面伙伴分配器统计信息

以 Ubuntu 为例列举 CPU 性能分析常用命令

发表于 2017-02-09   |   分类于 performance   |     |   阅读次数

操作系统版本 Ubuntu 16.04.1

手工实践记录，把新知识手工敲一遍，是把知识变成自己的第一步

uptime

查看平均负载

vmstat

包括系统范围的平均负载

虚拟内存统计信息命令，vmstat(8)，在最后几列打印了系统全局范围的 CPU 评价负载，另外在第一列还有可运行线程数

输出的第一行是系统启动以来的总结信息，r 列除外，这个是显示当前值的

r 运行队列长度 – 可运行线程的总数
us 用户态时间
sy 系统态时间
id 空闲
wa 等待IO，即线程被阻塞等待磁盘 IO 时的 CPU 空闲时间
st 偷取（未在输出里显示），CPU 在虚拟化环境下载其他租户上的开销

mpstat

单个 CPU 统计信息

多处理器统计信息工具，能够报告每个 CPU 的统计信息

-P ALL 用来打印每个 CPU 的报告，默认只打印系统级别的总结信息

CPU 逻辑 CPU ID，或者 all 表示总结信息
%usr 用户态时间
%nice 以 nice 优先级运行的进程用户态时间
%sys 系统态时间（内核）
%iowait IO 等待
%irq 硬件中断 CPU 用量
%soft 软件中断 CPU 用量
%steal 耗费在服务其他租户的时间
%guest 花在访客虚拟机的时间
%idle 空闲

sar

历史统计信息

-P ALL 与 mpstat 的 -P ALL 选项相同

-u 与 mpstat(1) 默认输出相同，仅包括系统范围的平均值

-q 包括运行队列长度列 runq-sz 等待数加上运行数，与 vmstat 的 r 列相同和平均负载

ps

进程状态
进程状态信息，ps(1) 列出所有进程的细节信息，包括 CPU 用量统计信息

BSD 风格

a 列出所有用户
u 面向用户的扩展信息
x 没有终端的进程 终端在 tty 一列显示

SVR4 风格

-e 列出所有进程
-f 列出进程的详细信息

CPU 用量的主要列信息是 TIME 和 %CPU
TIME 列显示了进程自从创建开始消耗的 CPU 总时间（用户态+系统态），格式为 “小时:分钟:秒”
在 Linux 上, %CPU 列显示了在前一秒内所有 CPU 上的 CPU 用量之和。一个单线程的 CPU 进程会报告 100%。而一个双线程的 CPU 型进程会报告 200%

top

监控每个进程/线程 CPU 用量

顶部是系统范围的统计信息，而下面是进程/任务的列表，默认按照 CPU 用量排序。系统范围的总结信息包括平均负载和 CPU 状态 %us %sy %ni %id %wa %hi %si %st 这些状态和 mpstat 相同，而与前面描述的一样，是所有 CPU 的平均值

CPU 用量通过 TIME 和 %CPU 列显示，与前面介绍的 ps(1) 里面相同

这个例子显示了名为 TIME+ 的列，和上面显示的一样，不过精度达到了百分之一秒。例如 “1:36.53” 代表在 CPU 上的时间总计为为 1 分 36.53 秒。有些版本的 top(1) 提供一个可选的 累计时间 模式，包括了已退出子进程的时间。

top 对 /proc 拍快照，会错过一些寿命较短的进程，这些进程在拍照之前已经退出。这在软件构建时经常出现，此时 CPU 被许多构建过程中短寿命的工具牢牢占据。Linux 上有一个 top 的变种 atop，可以捕捉短寿命进程。

pidstat

每个进程/线程 CPU 用量分解

按进程或线程打印 CPU 用量，包括用户态和系统态时间的分解。默认情况下，仅循环输出活动进程的信息。

上述例子捕获到了系统备份，包括了 tar 命令，从文件系统读取文件，以及 gzip 命令进行压缩。
gzip 的用户态时间较高，符合预期，其压缩代码为 CPU 密集型。
tar 命令从文件系统里读取，在内核中消耗更多时间

-p ALL 可以用来打印所有进程，包括空闲进程
-t 打印每个线程的统计信息

time

给一个命令计时，带 CPU 用例分解

使用 cksum 计算较大文件的校验码

第一次运行花了 5.1s，其中 2.8s 花在用户态 – 计算校验码，以及 0.3s 是系统时间
用来读取文件的系统调用。还有 2.0s 不见了 (5.1-2.8-0.3) ，很可能是花在等待磁盘 IO 读上，因为这个文件只有部分被缓存。

第二次运行完成得快很多，2.5s，几乎没被阻塞在 IO 上，这符合预期，因为文件可能在第二次运行时被完全缓存起来了

Linux 上，/usr/bin/time 版本支持下面的详细信息

DTrace

强大的动态追踪工具，可以和 SystemTap 之间相互转化，支持 SmartOS 及 FreeBSD 系列，这里省略

perf

CPU 性能计数器分析

ubuntu 16.04.1 安装 perf

perf 子命令

annotate 读取 per.data (由perf record 创建) 并显示注释过的代码

系统剖析

perf 可以用来剖析 CPU 调用路径，对 CPU 时间如何消耗在内核和用户空间进行概括总结。这项工作又 record 命令完成，该命令以一定间隔进行取样，并导出到一个 perf.data 文件，然后使用 report 命令查看文件。

-a 采样所有 CPU
-F 997 以 997HZ 频率
-g 采样调用栈
sleep 10 采样持续 10s

stat

stat 命令基于 CPC 为 CPU 周期行为提供了一个概要总结。

其它一些 Linux CPU 性能工具

htop 包括了 CPU 用量的 ASCII 柱状图，比最初的 top 更强的交互模式
